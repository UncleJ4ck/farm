## the challenge

There was no real session backend. The app simulated one with a cookie named `PHPSESSID` that held a base64-encoded, serialized `PageModel` object. On each request it base64-decoded the cookie and passed it to `unserialize()`. An autoloader pulled in any class whose name ended in `Model` from the `models/` directory.

`PageModel` had a `__destruct()` that called `include()` on its `file` property. That method runs automatically when the object is destroyed at the end of the script.

## the bug

The cookie value is attacker-controlled and goes straight into `unserialize()`, so I control the resulting object and its `file` property. When the script ends, `__destruct()` runs `include($this->file)`. Controlled path into `include()` is local file inclusion.

Decoding the default cookie showed the serialized shape:

```text
O:9:"PageModel":1:{s:4:"file";s:15:"www/index.html";}
```

I rebuilt it with a traversal path to prove the LFI:

```text
O:9:"PageModel":1:{s:4:"file";s:25:"../../../../../etc/passwd";}
```

base64 that, set it as `PHPSESSID`, and the response included `/etc/passwd`.

## the solve

LFI to RCE went through log poisoning. nginx writes requests to `/var/log/nginx/access.log`, including the `User-Agent`. I generated the cookie pointing the include at that log:

```php
<?php
    class PageModel
    {
        public $file;
    }
    $page = new PageModel;
    $page->file = '/var/log/nginx/access.log';
    print(base64_encode(serialize($page)))
?>
```

First I sent a request with PHP code in the `User-Agent`, which nginx logged verbatim. Then I sent the poisoned cookie so the app included the access log. The included log got parsed as PHP and my injected code ran. A jsonpickle variant of the encoder is in my notes but it produces the wrong serialization format and was a dead end, the PHP `serialize()` output is what the target expects.

## the flag

With code execution through the poisoned log, I read the flag file and it came back rendered in the page. It read like `HTB{...P0i5on...}`.
